# RISK_REGISTER

| ID | Risk | Type | Current Impact | Mitigation | Status |
| --- | --- | --- | --- | --- | --- |
| R-001 | Bun is installed locally but not exposed on the current non-interactive shell `PATH`. | toolchain | Local command execution still needs an explicit `PATH` prefix until shell startup config is reloaded. | Use `export PATH="$HOME/.bun/bin:$PATH"` in automation and document the requirement. | mitigated |
| R-002 | The root workspace originally had no git repository, and cloned external references under `ref/dashboard/*` could have become accidental embedded repos in a future upload. | process | The root workspace is now initialized as a git repository and the cloned reference repos are ignored so they do not become accidental embedded submodules. | Keep the root repo as the only tracked repository for framework delivery; continue ignoring `ref/dashboard/ToolJet`, `ref/dashboard/frappe`, and `ref/dashboard/metabase` unless they are intentionally converted to submodules later. | mitigated |
| R-003 | The required scope is extremely large and spans platform, security, DB, UI, bundles, connectors, migrations, and multiple verticals. | delivery | Risk of silent incompleteness if tracking is not rigorous. | Maintain exhaustive task matrix and keep unchecked items visible until code + tests + commands exist. | open |
| R-004 | Postgres integration tests depend on local services being available. | environment | Live Postgres verification now works through either generated Docker assets or a resident local server, but the Docker daemon is not running in this environment. | Keep both verification paths documented (`docker compose` and local server), and prefer the CI service-container path for deterministic automation. | partial |
| R-005 | Release signing uses a checked-in dev verification key locally and environment-backed key material in CI. | supply-chain | The local path is now fully testable, but the dev key must never be confused with production signing material. | Keep CI release readiness dependent on environment signing secrets and document the dev key as verification-only. | partial |
| R-006 | Representative implementations for every mandatory domain base may drift toward placeholder quality if not standardized. | architecture | Weak domain packages would violate the spec’s “no fake implementation” bar. | Use shared generator patterns and require each plugin to include a real manifest, resource, action, service, policy, and tests. | open |
| R-007 | The generated scaffolds will not typecheck until the core platform packages export the documented DSLs and helpers. | integration | This previously blocked workspace verification. | Core packages are now implemented and verified with root typecheck + test runs. | mitigated |
| R-008 | Many generated installable plugins still use baseline service logic and light contract tests rather than deeper persistence/workflow integrations. | delivery | The foundation wave plus CRM/sales/marketing and commerce/catalog/pricing/payments are now meaningfully hardened, but several remaining domain, feature-pack, and vertical packages still need richer behavior before the full “production-grade baseline” bar is met. | Harden plugins phase-by-phase, starting with foundations and the most critical domains/verticals, and keep incomplete areas explicit in `TASKS.md` and `STATUS.md`. | open |
| R-009 | The GraphQL adapter currently uses JSON-scalar payload boundaries instead of fully synthesized typed object graphs. | API | GraphQL support works, but its public typing fidelity is lower than the long-term target. | Keep the current adapter source-of-truth aligned to REST and add richer type synthesis in a later milestone. | open |
| R-010 | PostgreSQL integration, RLS bootstrap SQL, and migrator/runtime role verification are now implemented, but end-to-end migration execution against Postgres is still shallow. | environment | Live Postgres role/grant/RLS behavior is covered, but cross-package migration execution and rollback behavior still lean on unit coverage more than full DB orchestration. | Keep the generated bootstrap assets and live integration suite in CI, then add deeper migration execution coverage in a follow-on milestone. | partial |
| R-011 | Booking concurrency is now guarded at the service/integration-test layer, but not yet by persistent DB constraints or transactional reservation tables. | data integrity | The current canonical writer proves the reservation policy in-memory, but a future persistence layer could still regress without unique indexes or ledgers. | Carry the booking concurrency contract into Postgres schema design with reservation ledgers or uniqueness constraints, then add DB-backed integration tests. | open |
| R-012 | Coverage generation runs without environment wiring by default, so env-bound Postgres integration coverage can still be skipped in ad hoc local runs. | quality | The clean-shell `ci:check` uses live Postgres and passes, but a plain `coverage:report` run without `TEST_POSTGRES_URL` still falls back to the environment-less path. | Keep live Postgres verification in CI and document the env-backed coverage path for local deep verification. | partial |
| R-013 | The framework/package layer is complete, but the remaining plugin/domain/vertical scope is intentionally deferred. | delivery | A future engineer could mistake the green framework gate for full product completion if the deferred scope is not called out clearly. | Keep plugin/domain/vertical work explicitly unchecked in `TASKS.md` and called out as deferred in `STATUS.md`. | open |
| R-014 | The admin desk currently uses a memory-backed preference store in the verification harness, while durable per-tenant persistence for favorites/recents/saved views/dashboards is still only a framework contract. | UX / persistence | The admin package surface is real, but the default desk still needs plugin-backed durable preference resources before product teams can treat those preferences as production data. | Keep the contract surface stable now, then add durable admin-shell-backed persistence in the following plugin/business wave. | partial |
| R-015 | The admin-desk verification reconciliation lagged the exact root gate while the long-running workspace checks completed. | verification | The exact clean-shell `ci:check` rerun is now green and the tracking docs have been reconciled, so there is no remaining framework/admin verification blocker. | Keep future desk waves held to the same exact root-gate recording discipline. | mitigated |
| R-016 | Canonical alias wrappers and legacy `ui-*` compatibility layers can drift if future changes land in only one layer. | architecture | Admin-plugin work is now correctly pointed at the canonical wrappers, but regressions could still appear if maintainers change only the compatibility layer or only the alias surface. | Keep alias packages under unit coverage, keep the raw-import policy scan active, and prefer new admin-facing behavior in the canonical packages first. | partial |
| R-017 | The built-in AI packs currently expose durable runtime, memory, and eval operator flows through deterministic service fixtures rather than a full persisted control plane. | architecture / delivery | The framework contracts, admin surfaces, and CLI are in place, but production-grade long-lived state management still needs deeper DB/workflow binding. | Keep the framework-facing types stable now, then move built-in AI pack services onto persistent resources/jobs/workflows in the next AI hardening wave. | partial |
| R-018 | MCP interoperability is currently descriptor-first: the framework can derive governed tool/resource/prompt descriptors and inspect them through the CLI, but it does not yet run a full long-lived transport server or an external MCP connector fleet. | interoperability | The framework is ready for MCP-safe contract generation, but not yet for full remote/server lifecycle management. | Treat descriptor generation as the stable baseline, then add long-running transport serving and governed external MCP connectors as a dedicated follow-on milestone. | partial |
| R-019 | Eval datasets, baselines, and regression gates exist, but the root CI/release pipeline does not yet fail automatically on eval regressions above policy thresholds. | quality / release | AI quality can be measured and compared today, but release enforcement is still manual/documented rather than mandatory at the top-level gate. | Add eval-threshold execution and gate evaluation into the root CI/release scripts once the persistent AI control plane is wired. | partial |
| R-020 | AI provider access is intentionally wrapper-first, but representative provider connectors for live model invocation, embedding pipelines, and governed failover are not yet implemented in the plugin connector layer. | integration | The framework prevents raw SDK sprawl, but live provider posture is still limited to the contract layer and deterministic plugin fixtures. | Implement governed AI provider connectors with host/secret/trust policies before calling the AI platform fully provider-complete. | open |
| R-021 | The understanding layer now guarantees package-level doc packs repo-wide, but many legacy resources, fields, and actions still rely on warning-level rather than fully curated semantic metadata. | AI comprehension / documentation | AI and human readers now get a universal baseline, but some older packages still provide shallower inline meaning than the ideal long-term standard. | Keep `docs:validate` in the root gate, keep the generated understanding index fresh, and backfill richer semantic metadata in bounded sweeps starting with the most business-critical packages. | partial |
